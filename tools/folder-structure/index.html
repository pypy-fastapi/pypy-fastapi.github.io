    Folder structure

    Quote

    FastAPI is a powerful, modern web framework, but as a beginner, it can feel unstructured compared to opinionated frameworks like .NET or Spring Boot, which enforce a standard project architecture.

    üí° Why FastAPI Feels "Unstructured"

    FastAPI is a microframework:

    • It gives you freedom to structure your project however you want.
    • But that also means you have to decide where to put routes, services, models, etc.
    • This is very flexible, but it lacks the "starter scaffolding" that .NET or Spring Boot provide.

    üèóÔ∏è In .NET or Spring Boot:

    You usually see this kind of structure right away:

    src/
‚îú‚îÄ‚îÄ controllers/
‚îú‚îÄ‚îÄ services/
‚îú‚îÄ‚îÄ models/
‚îú‚îÄ‚îÄ repositories/

    That‚Äôs automatically enforced by templates or conventions.

    Aspect .NET / Spring Boot FastAPI
    Folder structure Enforced by framework Up to the developer
    Learning curve Steeper but guided Easier to start, harder to scale
    Flexibility Limited by convention Very flexible

    In real-world FastAPI projects, folder structure can vary depending on the application size, team preferences, and complexity. However, there are a few common architectural patterns used when organizing folders in FastAPI projects.

    Here are the main folder structure patterns:

    Structure:

‚îÇ   ‚îú‚îÄ‚îÄ api/           # Routers (View layer)

    ‚ùå Cons:

    üê≥ 5. Microservices Structure

    üìå Summary Table

    Pattern Best For Complexity Scaling Notes
    Flat/Monolithic Small apps, MVPs üü¢ Low üî¥ Low Fast to build, slow to grow
    Layered (MVC-like) Medium apps, structured devs üü° Medium üü° Medium Familiar and maintainable
    Modular (Feature-based) Large teams, feature teams üü† Medium+ üü¢ High Clean isolation by feature

    Microservices Distributed systems üî¥ High üü¢ High Requires infrastructure setup

    üë£ Recommendation

    • Refactor to Modular when features grow
    • Adopt DDD/Microservices only when justified by scale

    ‚úÖ Why This Structure Is Most Popular

    • Scales well for small to large projects
    • Keeps business logic separate from DB logic and API routing
    • Makes testing, dependency injection, and auth easier
    • Easily supports versioned APIs
    • Plays nicely with Docker, Celery, Alembic, Pytest, etc.

    üß± Tips for Real-World Use

    • Use Pydantic models in schemas/ for all request/response shapes.
    • Use SQLAlchemy or SQLModel in models/.
    • Keep business rules in services/, not in the route handler.
    • Handle auth and user context via dependencies/.
    • Add versioning (e.g., /api/v1, /api/v2) for long-term maintainability.
    • Use core/config.py with pydantic.BaseSettings to manage .env and settings.
